# Database Architecture

This document defines the database architecture and design principles used in Project NOAH.

This document serves as the reference for designing, deploying, and maintaining the database layer throughout the project.

---

## Purpose

The database layer is responsible for storing persistent data generated by Project NOAH.

The database should provide a reliable foundation for long-term memory, application data, and future services.

Examples include:

- User information
- AI memory
- Workflow metadata
- n8n configuration
- Agent state
- Future application data

---

## Database Engine

Project NOAH uses PostgreSQL as the primary relational database.

### Current Version

- PostgreSQL 17

### Deployment

- Docker Compose

### Container

- noah-postgres

### Default Port

- 5432

---

## Why PostgreSQL

PostgreSQL was selected because it provides:

- High reliability
- ACID compliance
- Excellent Docker support
- Strong SQL standard compatibility
- JSON / JSONB support
- Long-term scalability
- Well-suited for future AI and data-intensive workloads

---

## Database Naming Convention

### Database

```text
noah
```

### Container

```text
noah-postgres
```

### Docker Volume

```text
noah-postgres-data
```

### Docker Network

```text
noah-network
```

Future database resources should follow the same naming convention.

Consistent naming simplifies infrastructure management and maintenance.

---

## Data Storage Policy

Database data must never be stored inside containers.

Persistent data is stored using Docker Volumes.

Database storage should remain independent of container lifecycles.

Current volume:

```text
noah-postgres-data
```

Benefits:

- Safe container recreation
- Reliable backups
- Easy upgrades
- Persistent storage

---

## Connection Policy

Application services communicate through the Docker network.

Internal communication should remain isolated within the Docker network whenever possible.

External applications should connect only through explicitly exposed ports.

---

## Security Policy

Sensitive information must never be committed to Git.

Examples include:

- Database passwords
- API keys
- Tokens
- Secrets

Environment variables are managed using:

```text
compose/.env
```

The `.env` file is ignored by Git.

---

## Backup Strategy

Future backup strategy should include:

- Scheduled PostgreSQL dumps
- Docker Volume backups
- Off-site backup support
- Restore validation

---

## Future Database Services

The database layer may be expanded with:

- Redis
- Qdrant
- MinIO
- pgvector
- Additional PostgreSQL databases (if required)

Additional services should only be introduced when they provide clear architectural value.

---

## Upgrade Policy

Project NOAH prioritizes stable software versions.

Upgrades should prioritize stability over adopting the latest version.

Major version upgrades should be performed only after compatibility verification.

Example upgrade workflow:

- PostgreSQL 17 → PostgreSQL 18
- Backup validation
- Restore testing
- Migration verification

---

## Design Principles

Project NOAH follows these principles:

- Stability over novelty
- Documentation before implementation
- Reproducible infrastructure
- Docker-first deployment
- Secure configuration management
- Long-term maintainability
